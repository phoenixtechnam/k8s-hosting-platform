# ADR-029: Secrets management + disaster recovery for a public OSS platform

**Status:** Accepted · 2026-04-22
**Supersedes:** none
**Related:** ADR-028 (backup architecture), ADR-022 (external DNS/IAM/VPN services)

## Context

This platform targets a public OSS deployment model: operators clone the repo, run `bootstrap.sh`, and stand up their own cluster. That imposes two constraints on secrets handling the in-git-encrypted-secrets pattern (e.g. committed SealedSecrets) doesn't gracefully satisfy:

1. **No secret material in the public repo.** Every per-deployment key/password/token must be generated or provided by the operator at install time, not committed.
2. **Operator-held recovery key.** The "I can restore from zero" guarantee must depend on something the operator (not us) controls — their own asymmetric key — so that losing the cluster does not strand their data.

A failed-server recovery walks through four classes of artifact:
- **Ephemeral**: session tokens, in-memory queues, websocket connections — acceptable loss
- **Derivable from code**: all k8s manifests, container images, `cert-manager` `Certificate` CRs — rebuilt from git + GHCR
- **Data**: platform DB + tenant PVCs — recoverable from Longhorn/S3/SSH backups (verified 2026-04-22)
- **Deployment secrets**: `OIDC_ENCRYPTION_KEY`, JWT secret, DB password, admin seed, OAuth2 client secret, TLS material — **these are the class this ADR addresses**

## Decisions

### 1. age + SOPS over SealedSecrets

We use [`age`](https://github.com/FiloSottile/age) v1.3.1 (actively maintained, last release 2025-12-28, 22k+ stars) as the primary secret-encryption primitive for backup artefacts. age is preferred over:

- **SealedSecrets** — the controller's private key lives in the cluster. To survive a cluster loss we'd have to back up that private key anyway; the controller just adds another moving part.
- **GPG** — age is specifically designed to replace GPG for file encryption. Smaller key format, no web-of-trust baggage, no subkey confusion.
- **KMS (AWS/GCP/Azure)** — couples every OSS installer to a paid cloud service; anti-goal for this project.

**How we use it:** a nightly CronJob enumerates cluster Secrets, produces a tar.gz of YAML manifests, encrypts the tarball to the operator's age public recipient, and uploads to the active backup target (S3 or SSH — same target Longhorn uses for volume backups). No secret material ever lands in git.

**SOPS** is mentioned in this ADR's title because Flux's native SOPS integration is the canonical way to commit encrypted secrets to git if we ever need to (configuration drift, audit, declarative reconciliation). Today's design does NOT commit encrypted secrets to git — the backup bucket holds them — but leaves that door open.

### 2. Operator key ownership: single recipient by default, multi-recipient supported

`bootstrap.sh` prompts for an age public recipient (`age1…`). If none is supplied it runs `age-keygen`, prints the private key ONCE to stderr wrapped in a highly visible "SAVE THIS NOW" banner, and stores only the public recipient in a cluster `ConfigMap` (`platform-operator-recipient`). Operators can supply multiple recipients (`--operator-age-recipient=ageA,ageB`) so a team has independent recovery paths.

**Loss of the private key = cluster is unrecoverable from scratch.** This is intentional and documented — the same property applies to losing the Hetzner account password or the DNS registrar credentials. No automation can rescue an operator who has no off-box secret material.

### 3. Per-deployment secrets are generated by `bootstrap.sh`, never committed

The following are generated on first install and live only in-cluster (plus the encrypted backup bundles):
- `OIDC_ENCRYPTION_KEY` (32 bytes, AES-256-GCM key for DB-column encryption)
- `JWT_SECRET` (32 bytes, HMAC key for session tokens)
- `PLATFORM_INTERNAL_SECRET` (shared-secret for pod-to-pod trust)
- `DB_PASSWORD` (postgres superuser)
- `ADMIN_SEED_PASSWORD` (printed once; operator rotates via admin panel)
- OAuth2 proxy client secret (generated during Dex client registration)

**The public repo ships the *scheme*, not the *values*.** Example configs use placeholders; operators never copy-paste secrets from the repo.

### 4. Backup target supports both S3 and SSH

The `backup_configurations` table + admin-panel UI already accept both modes. The reconciler and DR CronJobs (etcd, pg-dump, cluster-state, secrets-backup) branch on the active config's `storage_type`:
- **S3** — `aws s3 cp` via `amazon/aws-cli` image
- **SSH** — `rsync` via a scratch image with `openssh-client`, authenticating with the decrypted SSH private key mounted from `backup-credentials`

Single operator deploying to their own VPS: SSH is fine.
Operator deploying to Hetzner Cloud: S3 (Object Storage) is fine.
Operator with both: active target picks which one today, switchable any time via admin panel.

### 5. Tenant PVC backup is opt-in by label, audited daily

Every tenant PVC is labeled `recurring-job-group.longhorn.io/default=enabled` at creation time (backend enforcement, one-shot migration script for pre-existing). A daily audit CronJob emits a Kubernetes `Warning` event for any Longhorn PVC lacking the label — unless it carries the opt-out annotation `platform.phoenix-host.net/backup-excluded=true`.

This closes the silent-footgun where a forgotten label meant a tenant's data was quietly not backed up.

### 6. What is and isn't in scope

**In scope of this repo's DR story:**
- Platform DB (PVC + logical `pg_dumpall --globals`)
- Tenant PVCs (Longhorn)
- Platform control Secrets + tenant namespace Secrets + cert-manager TLS secrets (new secrets-backup CronJob)
- K8s etcd snapshot
- Cluster-state YAML inventory
- hostPath `/var/lib/platform/snapshots` (storage-lifecycle tarballs)
- Stalwart mail PVC (when Stalwart deploys)

**Out of scope — operator responsibility:**
- PowerDNS zones (covered by the DNS project per ADR-022)
- Hetzner Cloud firewall rules (Terraform module can be published separately)
- DNS registrar credentials, Hetzner account access, operator age private key

## Consequences

**Wins:**
- Public OSS distribution without leaking per-deployment secrets
- Single recipient model is simple enough for solo operators; multi-recipient option exists for teams
- Same operator key decrypts backups on any future restored cluster
- Backup target is operator-choice (S3 or SSH), no vendor lock-in

**Trade-offs:**
- Operator MUST secure the age private key themselves. Lose it and the cluster is unrecoverable. This is a hard constraint, not a bug.
- First-boot is more interactive than a fully silent install because of the key-generation prompt. Non-interactive installs must pass `--operator-age-recipient`.
- The restore procedure is a runbook, not a one-button action. We publish `scripts/dr-restore.sh` but it still expects the operator to supply the private key and the restore-from URL.
